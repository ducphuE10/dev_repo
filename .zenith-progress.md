# Zenith Progress Log

## Codebase Patterns
<!-- Consolidated knowledge from all iterations. Read this FIRST. -->
- `.zenith-progress.md` is the canonical append-only implementation log; do not reintroduce `PROGRESS.txt`.
- The intended stack is TypeScript throughout with pnpm workspaces and Turborepo coordinating `apps/api`, `apps/mobile`, `apps/web`, and shared `packages/*`.
- The workspace namespace is `@dupe-hunt/*`, with root-level `pnpm lint`, `pnpm typecheck`, and `pnpm test` serving as the stable quality entry points.
- Early bootstrap verification lives under `scripts/`, while app-level `dev` commands can stay as lightweight long-running stubs until the framework-specific implementations land.
- Shared object contracts belong in `packages/types`, shared database code belongs in `packages/db`, and apps must not import directly from each other.
- Shared TypeScript defaults live in `packages/config/typescript/base.json`, with the root `tsconfig.json` extending that base and owning workspace-specific `paths` mappings.
- Environment contracts are centralized in `packages/config/src/index.ts`, while each app keeps a matching `.env.example` file as the source-of-truth template for local setup.
- `packages/db` owns the schema layer and exports a `createDatabaseClient` helper plus Drizzle table definitions from TypeScript source without requiring a build step.
- Database migrations live as ordered SQL files in `packages/db/migrations`, and the repo-standard way to apply them is `pnpm db:migrate`, which delegates to `packages/db/scripts/migrate.mjs`.
- The migration runner resolves `DATABASE_URL` from `apps/api/.env`, root `.env`, or the active shell, so local infra and API config stay aligned.
- Local infrastructure is defined in `docker-compose.yml` with PostgreSQL 16 and Redis 7 defaults that match the API example environment file.
- Docker Compose host ports are overridable with `POSTGRES_PORT` and `REDIS_PORT`, which keeps the documented local workflow usable on machines where `5432` or `6379` is already taken.
- Source `.ts` files that run directly under Node 24 should use explicit `.ts` relative imports, and the shared TypeScript config must keep `allowImportingTsExtensions` enabled to match that runtime model.
- Always use `IF NOT EXISTS` for migration SQL statements.
- Fastify API routes are expected to use Zod validation, consistent `{ error: { code, message } }` responses, Drizzle for database access, and BullMQ for heavy async work.
- The API now isolates external auth and database reads/writes behind injected `authProvider` and `repository` boundaries, which keeps Fastify integration tests deterministic without live Supabase or Postgres dependencies.
- Protected API routes verify bearer tokens locally with `JWT_SECRET`, then hydrate the app-level user profile from the `sub` claim before serving `/users/me*` flows.
- Post creation now flows through `app.repository` for persistence and `app.jobs` for enqueue-only background work, which keeps HTTP handlers synchronous while deferring video and affiliate processing to later worker iterations.
- Feed pagination uses the last returned `post.id` as the cursor, and each tab must keep a deterministic tie-break order of primary rank metric, then `created_at`, then `id`.
- Mobile screens should stay thin: screen files in `screens/`, reusable UI in `components/`, business logic in custom `hooks/`, React Query for server state, and SecureStore only for auth tokens.
- Next.js web pages are browse-only in v1, prefer server-side data fetching for SEO, and must provide metadata for every public route.
- Feed queries must always filter `status = 'active'` and never return flagged or removed posts.
- Affiliate click tracking happens server-side in the redirect endpoint, never in the frontend.
- Social interaction routes now persist votes/saves/flags directly in the repository layer while still enqueuing `update-post-counts` and `auto-flag-review` jobs so worker-backed processing can replace the synchronous path later.
- Internal moderation routes are protected by the shared `x-admin-key` header, which must match `ADMIN_API_KEY`.
- Search queries are normalized to lower-case single-space terms, and trending terms are stored in the Redis sorted set key `search:trending`.
- Receipt images live in the private R2 bucket only and must never be exposed to clients.
- Background worker jobs handle heavy async work such as video processing, OCR, affiliate wrapping, and aggregation counter updates.
- The Expo app should avoid a `src/app` directory unless Expo Router is intentionally enabled, because Expo will otherwise auto-detect that path as a router root.
- Mobile preview sessions should not send placeholder bearer tokens to optional-auth API routes, because invalid JWTs turn otherwise-public feed requests into `401` responses.
- The mobile auth provider should own refresh-on-hydrate plus `/users/me/categories` sync so screens only consume session state instead of juggling tokens or onboarding persistence.
- The mobile feed should use React Query infinite queries keyed by tab and category slug, and the dupe detail screen should live on a nested Home stack above the tab navigator.
